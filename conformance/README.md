# Charter Conformance Suite

> Language-neutral test vectors that pin Charter behavior. Any implementation
> of the protocol — in any language — is "conformant" iff it passes every
> vector under [`vectors/`](vectors/) against the rules in
> [`SPEC.md`](SPEC.md).

This subdirectory is the **single source of truth** for cross-language
conformance. It does NOT replace the Python reference implementation's
own `tests/` suite — that's still where Pythonic edge cases and
adversarial scenarios live. Conformance vectors are the strict subset of
behavior that EVERY implementation owes.

---

## What's here

```
conformance/
├── README.md                 ← this file
├── SPEC.md                   ← the rules (anchor target for every vector)
├── vectors/                  ← 30+ JSON fixtures, organized by topic
│   ├── sign/                 ← canonical bytes + signing
│   ├── verify/               ← signature verification (valid + tampered)
│   ├── aggregate/            ← TYPE_TO_DECISION + aggregate precedence
│   ├── chain/                ← string-based chain verification
│   ├── lifecycle/            ← state machine
│   ├── jwks/                 ← JWK derivation + cross-check + pins
│   ├── transparency/         ← SHA-256 chained log
│   ├── privacy/              ← SD-JWT path 1 redaction
│   └── generation_metadata.json   ← when + which charter rev built them
├── generators/
│   └── build_vectors.py      ← regenerates everything from charter Python API
└── runners/
    ├── python/run.py         ← full Python reference runner
    ├── javascript/{run.mjs, package.json}  ← skeleton; 1 vector demo
    └── rust/{Cargo.toml, src/main.rs}      ← skeleton; 1 vector demo
```

## Running the Python runner (the authoritative one)

From the repo root:

```bash
# every vector, summary to stdout
python conformance/runners/python/run.py --vectors-dir conformance/vectors

# subset via substring filter
python conformance/runners/python/run.py \
    --vectors-dir conformance/vectors \
    --filter aggregate

# emit JUnit XML for CI integration
python conformance/runners/python/run.py \
    --vectors-dir conformance/vectors \
    --junit-xml conformance-results.xml
```

Exit code is 0 iff every vector passes; non-zero on any failure. CI relies
on this (see `.github/workflows/ci.yml`).

## Running the JS / Rust skeletons (1 vector each)

```bash
# JS (Node >= 18)
cd conformance/runners/javascript
node run.mjs

# Rust (cargo >= 1.70)
cd conformance/runners/rust
cargo run --release
```

These are **proofs that the framework is implementable**, not full
runners. The other ~29 vectors are TODO until B1.2 (JS SDK) and a
future Rust SDK ship — at which point they take over execution.

## Adding a new vector

1. Pick the right subdirectory under `vectors/` (matching the SPEC
   section it tests).
2. Add a generator function to `generators/build_vectors.py` that calls
   the charter Python API, captures the result, and writes a vector
   JSON conforming to the schema in [`SPEC.md` Appendix A](SPEC.md#appendix-a--vector-json-schema).
3. Run `python conformance/generators/build_vectors.py` to regenerate.
4. Make sure `python conformance/runners/python/run.py` still passes.
5. Reference the matching SPEC.md anchor in the vector's `spec_section`.

> **Never hand-edit vector `expected_output` fields.** Vectors are
> generated artifacts. A change to expected behavior must come from a
> code change to charter + an explicit SPEC.md version bump, then be
> picked up by regenerating.

## Writing a new runner (in a new language)

A conformant runner:

1. Reads every JSON file under `vectors/**/*.json` and parses the schema
   in [SPEC.md Appendix A](SPEC.md#appendix-a--vector-json-schema).
2. Dispatches each vector to the appropriate handler based on its
   parent directory (`sign/` → canonical-bytes/sign roundtrip,
   `verify/` → verify, etc.).
3. Compares the implementation's output against `expected_output`
   (when non-null), or asserts the named exception was raised
   (when `expected_error` is non-null).
4. Emits a summary line per vector (`PASS` / `FAIL`) and exits 0 iff
   every vector passed.

Optionally emits JUnit XML for CI consumption.

See `runners/python/run.py` for the reference implementation.

## How vectors stay correct

- Vectors are regenerated by `build_vectors.py` from the Python API.
- Once committed, `expected_output` is **frozen** — any divergence in
  Python output is treated as a regression, and CI catches it because
  `tests/test_conformance.py` runs every vector against the current
  Python code.
- A deliberate breaking change to behavior requires: (1) bump SPEC.md
  version, (2) regenerate vectors, (3) loudly call out the bump in
  CHANGELOG.md, (4) communicate to downstream implementers.

## Roadmap

- **v0.1 (this Issue)** — Initial 30+ vectors. Python runner authoritative;
  JS + Rust runners are skeletons.
- **v0.2 (after B1.2)** — JS runner runs the full vector set; becomes the
  second authoritative runner.
- **v1.0 (after multi-language production deployments)** — Spec frozen;
  vectors covered by signed releases; conformance dashboard at
  `charter.dev/conformance`.
