§ 2 — 协议地图

Agent 协议三层。中间是空白。

Agent 生态今天还在裸奔。

层 它回答什么 现状

Authority 我代谁做、不会做什么? 空白

§ 3 — 我们的答案

Charter 用 clauses 把规则写下来

一份签名 JSON。核心是 7 条自然语言条款,
每条带 id 和 type,确定性映射到 verdict。

Charter alice@acme.com × research_agent_v1 charter.dev/alice@acme.com/research_agent_v1
  1. C-001 scope 会计、报税、做账、财务分析、发票分类、税务凭证整理 → allow
  2. C-002 out_of_scope 营销文案、广告设计、代码编写、UI 设计 → incompatible
  3. C-003 approval_required 客户 PII(姓名 + 银行 / 税号 / 收入组合) → needs_approval
  4. C-004 approval_required DROP / DELETE / TRUNCATE / 删除生产备份 → needs_approval
  5. C-005 operational_limit Mon–Fri 09:00–18:00 NY · 单任务预算 $0.50 → needs_approval
  6. C-006 data_handling 客户报税资料、税号、银行流水 — 不外传、不持久化 → needs_approval
  7. C-007 style 结构化输出(JSON / 表格)· 引用来源 · 中英文均可 → allow

clause.type 通过协议常量 TYPE_TO_DECISION 确定性映射到 local decision。 多条命中时,按 incompatible > needs_approval > allow 聚合得到最终 verdict。

LLM 只判断"这条 clause 是否命中", 不允许决定 verdict —— 协议层确定、可审计、可重放。

§ 5 — 下一步

v0 跑通了,下一步呢?

协议层往何处扩展 — 已设计未实现 / 路线图上的方向。

  • 01

    Charter Chain · 多跳 attenuation

    协议

    agent 派给 sub-agent 时,sub-Charter 必须严格 ⊆ 上游 Charter。OCR sub-agent 嵌在五层之下仍受最上游 principal 的红线 — 合规审计真正想要的东西。

  • 02

    Budget Control + AP2 Mandate 支付集成

    支付

    Charter 已携带 budget_per_task_usd / budget_monthly_usd。协议层在执行前估算成本,超额降级 needs_approval。配合 AP2 Mandate 做原子付款授权:Charter 管"能花多少",AP2 管"这一笔具体批准"。

  • 03

    隐私计算 · ZK / SD-JWT / TEE 传输与验证

    隐私

    ZK 证明"Charter 满足某条件"而不暴露 clause 内容;SD-JWT 选择性公开个别 clause;TEE 跑 projection,服务端看不到 wizard 原文;Web Bot Auth 在 HTTP 签名头携带 charter_url,transport 层即可闸门。

  • 04

    Transparency Log · 可审计历史

    审计

    类似 Certificate Transparency。每次签发 / 撤销 / 替换 Charter 写入 append-only 公开 log。第三方可问"这个 agent 过去 30 天有没有违反自己的 Charter",不需要信任 issuer 或 runtime。

  • 05

    Loopback Verification + Retry

    检查

    propose_within_scope 自动把 rewrite 丢回 check_compatibility 验证,prompt 演化 + temperature 退火(0.0 → 0.4 → 0.7),返回验证过的 RewriteProposal 或带尝试历史的 RewriteFailure。
